Why You Should Use Git for Version Control?

Git, the version control system developed by Linus in 2005, has evolved into the most popular and advanced open-source version control tool, but there are still a lot of companies and teams that still use SVN or CVS to version control their projects, and some companies do have some valid reasons for maintaining the status quo, but Git makes our development and collaboration more efficient in most scenarios.

There have been many articles and discussions about Git versus other version control tools for quite a while now, and the development scenarios we face today are a lot different than they were a few years ago, and each of these different version control tools has evolved, but the author has always maintained that Git is the most efficient tool available today and that this is due to top-level design thinking, so let’s take a look at “Why You Should Use Git for Version Control” today.

Overview

When we talk about the best version control system, we need to find a few rivals for Git to compare against, so that we can understand more clearly what the design philosophy behind Git brings to the table.

The following is a list of the most common commands used by the two tools to check out the repository, update the local repository, commit locally, and commit to the main repository.

While the two version control systems have a lot of equivalent features, the differences between the two systems show the different design choices that they make. For example, SVN does not support local commits, which is a completely different design philosophy from Git.

Before we dive into the design decisions of Git, let’s go back to the question of “why you should use Git for version control” and think about it in a different way - what do we want from a version control system and how do we define a better one? Linus, in a 2007 Google Talk, described three characteristics that a version control system must-have.

The development model in the version control system must be distributed.
The version control system must provide good enough performance support.
the version control system must guarantee the integrity of the files and provide protection against data loss or corruption.

The last two of these three points are easy to understand, and while many may wonder why performance is so important for a version control tool that is used infrequently on a daily basis, no one can resist a performance boost (especially if it is free).

Design

In the previous section, we covered the three characteristics that a version control system must meet: distribution, performance, and reliability.

A distributed approach that better fits the scenario of how a version control system works.
Performance improvements encourage us to do things less expensively.
Reliability to help us catch data loss or corruption due to disk failure in a timely manner.
These features are essential to a good version control system, and we’ll cover each of them in turn and how Git is designed and implemented to follow these rules.

Distributed

If you’ve been using Git as a version control tool since you were a developer, you may not appreciate the limitations of using a centralized version control tool, but let’s say we’re using a centralized version control system where all developers have to connect to the same server in order to develop and commit.

Choosing a centralized version control tool means that we must accept several disadvantages.

Engineers must all need to be connected to the network in order to develop, and cannot commit in case of poor or no network conditions.
many people may think they don’t have the need to work offline, but this is actually more common in our daily work. 100 megabytes of bandwidth is sufficient for daily development, but in complex network environments, there are still many times when we encounter scenarios where the network is unavailable or extremely poor, such as on planes and trains.
for a larger distributed development team, it is also difficult for us to ensure that all members can connect to the same main repository at the same time through a high-speed network such as a backbone network in actual production.
that commits and changes to the central repository, such as branch creation, are visible to all developers.
when we use a centralized development model, whether we commit code or create new experimental branches, these operations actually change the codebase shared by all, which means that if a developer creates many experimental branches, the codebase for all developers becomes larger as well.
Although we can create branches in the central repository, since there is no namespace in the central repository, it is very easy to have naming conflicts if developers create branches that do not follow specific naming rules, such as various test branches.
all developers in the current repository need to have permission to commit code directly to the main repository, otherwise, they cannot develop.
Also giving all developers in the project write access is actually a dangerous thing; we don’t know if these developers have enough experience to operate the main repository, and if there is an operational error, all members will be exposed to the risks associated with this error.
All three of these problems are caused by a centralized master repository, and they are problems that cannot be avoided in a centralized system. These shortcomings of a centralized version control system are very detrimental to the experience, which is why Git chose to use a distributed design philosophy to guide its implementation.

In a distributed version control system, all nodes are equivalent in implementation, and they can access all data in other repositories at once, as long as they have access. Different nodes can pull code from each other. We can use the git clone command to download all the data from one repository and create branches, changes, and commits locally, or we can choose to push new changes to other nodes or wait for other nodes to pull them.

While all nodes are equivalent in implementation, in practice we may also have one or more master repositories that help multiple developers synchronize their code. The relationship between the nodes is more like a forkable tree structure, and with a distributed model, as opposed to a centralized version control system.

Git allows us to commit locally to support offline work.
Git allows us to create branches locally and without the problem of namespace conflicts.
Git allows commits to be done via Pull Request without all developers having to write access to the main repository.
For example, all version numbers in SVN are serial incremental numbers, so there are no conflicts and it’s easier to communicate.

Although a distributed version control system can bring us various benefits, it does not mean that a centralized system does not work. On the contrary, a centralized version control system can work better in projects where the code needs to be tightly controlled because the company or project makes the decision that “the code needs to be very tightly controlled” and the resulting reduction in efficiency is acceptable, so it is really a trade-off between development efficiency and permission review.

Many companies and decision-makers may very easily underestimate the degree of ‘efficiency reduction’ that a centralized repository can bring because they don’t understand engineering and version control systems.

In practice, however, there is very little code that really needs to be tightly controlled. Most companies have very low-security internal code that is not only of very poor quality but also doesn’t work once it’s out of the company’s environment.

Performance

Performance is also critical to a version control system, but it’s more important to choose the right metrics to measure performance. Linus’ talk at Google Talk mocked SVN developers for choosing “branch creation” as an important performance metric for their version control system, and for advertising it on their website (the exact interface is not verifiable).

However, it is the performance of ‘merging branches’ that is more important to a version control system, because the ultimate goal of creating a new branch is to merge it back into the master branch, and if the performance of merging branches is so poor that each merge can take tens of seconds or even minutes, then this expensive cost will make developers wary of using branch merges, and many may choose to develop directly on the master branch because of the complexity and difficulty of merging branches because they don’t see how changing just one or two lines of code can cause an online accident!

Most developers today already use Git for version control on a daily basis, and branching and merging is a very common feature. This is because it is so easy and fast to merge multiple branches with Git, and it is possible to compare and merge two complex branches in a matter of seconds without waiting too long.

It’s also very fast to compare differences between two different versions and folders. As a distributed system, each Git repository has a full amount of change data, so we don’t need to go through the network to get information about the repository, and we can quickly pinpoint the changes that are causing problems compared to a centralized version control tool.

Reliability

Reliability is also critical to a version control system. While professional cloud providers may not always be able to guarantee the reliability of our data and prevent corruption or loss, we as users and developers need some mechanism to protect our code from corruption.

Git generates a 20-byte SHA-1 hash for each version commit, for example: e328029255d8f02909ec0cbc16cc74ef4b79e1d0. This hash represents the version number of the current commit, and it is calculated by combining the following data.

The source tree of the current commit.
The SHA-1 hash of the last commit.
The names, emails, and other information of the author and committer.
The message of the current commit.

Since GitHub detected a hash collision with SHA-1 in 2017, the Git community is considering using more complex hashing algorithms instead of SHA-1, such as SHA-256. Summary

Documentation and the toolchain are very important for a tool. Good documentation not only helps beginners get started quickly but also helps users solve most of their confusion and understand the design and implementation behind the tool. The integration of SourceTree and the IDE no longer requires us to use the command line (which is what this author prefers to do), and all of these things have helped make the Git ecosystem more mature and are important reasons why we should use Git.

Let’s briefly summarize some of the points made in the text.

Git is a distributed version control system that allows developers to work offline and commit locally, not only to avoid the risks associated with committing large amounts of code directly but also to help limit the amount of authorization we can give to the main repository and reduce the number of conflicts caused by namespaces.
Git chooses to merge branches as a primary performance measure when optimizing performance, making it a very low-cost operation to encourage branching.
Git uses SHA-1 hashing to ensure the reliability of the data in the repository. We can use SHA-1 to verify the data and ensure the stability and reliability of all the data in the commit chain, which also helps us defend against malicious tampering from attackers.
